# API 注释规范（Controller 层）

## 1. 目标

- 保持所有 Controller 接口注释风格一致。
- 降低协作成本，便于前后端、测试与运维快速理解接口。
- 让异步接口与同步接口的语义表达清晰可读。

## 2. 适用范围

- `workspace/backend/src/main/java/com/example/psu/controller` 下所有控制器方法。
- 新增接口必须遵循本规范；历史接口逐步按本规范收敛。

## 3. 注释结构（方法级）

每个对外接口方法都使用以下结构：

1. 接口用途（一句话）
2. 请求方法与路径
3. 入参
4. 返回
5. 说明（可选，写副作用、约束、异常语义）

推荐模板：

```java
/**
 * 接口用途（一句话）。
 * 请求方法与路径：POST /api/xxx（兼容 /api/v1/xxx）。
 * 入参：字段A、字段B，或 XxxRequest。
 * 返回：XxxResponse / 202 ACCEPTED / 200 OK（无 body）。
 * 说明：可选，补充约束、异步语义、状态流转等。
 */
```

## 4. 术语统一

- `数据库主键ID`：指表主键 `id`。
- `业务ID`：指业务唯一标识（例如 `psuId`）。
- `版本号`：统一称“版本号”，不要混写“版本/版次”等。
- `异步接口`：统一写“返回：202 ACCEPTED”。
- `同步删除接口`：统一写“返回：200 OK（无 body）”。
- `兼容路径`：统一写“兼容 /api/v1/...”。

## 5. 同步/异步接口说明规则

- 同步接口注释要明确“同步”语义，返回业务结果体。
- 异步接口注释要明确“异步”语义，仅表示任务已提交。
- 若异步接口结果需要后续查询，在“说明”中写明推荐查询入口。

## 6. 示例

同步示例：

```java
/**
 * 创建测试数据集（同步）。
 * 请求方法与路径：POST /api/test-datasets（兼容 /api/v1/...）。
 * 入参：psuId + TestDatasetCreateRequest。
 * 返回：创建后的 TestDataset。
 */
```

异步示例：

```java
/**
 * 创建测试数据集（异步）。
 * 请求方法与路径：POST /api/test-datasets/async（兼容 /api/v1/...）。
 * 入参：psuId + TestDatasetCreateRequest。
 * 返回：202 ACCEPTED。
 * 说明：调用方可通过历史查询接口追踪处理结果。
 */
```

## 7. 维护建议

- 新增接口时先写注释再写方法体，避免遗漏文档。
- 代码评审时将“注释规范符合度”作为必查项。
- 若接口协议变更，注释必须同提交一起更新。
