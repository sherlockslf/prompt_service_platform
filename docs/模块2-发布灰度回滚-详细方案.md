# 模块2：对外提示词服务（发布/灰度/回滚）详细方案

## 1. 文档目标

- 将“版本审核能力”升级为“可生产发布能力”
- 补齐灰度发布与版本回滚闭环
- 提供可直接开发的数据库与 API 契约草案
- 作为后续研发任务拆解与联调基线

## 2. 背景与问题

当前项目已具备以下能力：

- 版本提审、审核、驳回
- 编排快照（revision）留痕
- 代码生成入口

当前缺口（必须补齐）：

- 没有“线上生效版本”模型
- 没有灰度策略与流量路由
- 没有可执行回滚机制
- 没有统一的对外解析服务接口

## 3. 目标与非目标

### 3.1 目标

1. 对外提供稳定的 Prompt 解析服务（按上下文返回应命中的版本）  
2. 支持发布单管理（创建、审批、执行、完成）  
3. 支持灰度规则（白名单、标签匹配、比例流量）  
4. 支持快速回滚到上一稳定版本并保留审计链路  

### 3.2 非目标（本阶段不做）

- 不做多云多活调度
- 不做复杂实验平台（A/B 统计平台后续迭代）
- 不做自动化 CI/CD 编译部署，仅完成发布域业务能力

## 4. 术语定义

- PSU：Prompt Service Unit，提示词服务单元
- Revision：编排快照版本（来自当前 `composition_revision_no`）
- Release：发布单，描述某 PSU 在某环境的一次发布动作
- Rule：灰度规则，决定请求路由到哪个 revision
- Stable：稳定版本，当前环境默认兜底版本
- Canary：灰度版本，规则命中后生效的候选版本

## 5. 角色与权限

- 业务：可发起发布申请、查看发布状态
- 研发：可审核发布、执行回滚、维护灰度规则
- 管理员：可强制终止发布、解除卡死状态
- 机器调用方：仅允许调用对外解析接口，无管理权限

## 6. 业务流程

### 6.1 发布主流程

1. 从“审核通过”的版本中选择目标 revision  
2. 创建发布单（目标环境 + 发布策略）  
3. 发布审核通过后执行发布  
4. 发布后写入稳定版本指针或灰度规则  
5. 发布完成后可通过解析接口被业务流量命中  

### 6.2 灰度流程

1. 发布单类型为 `CANARY`  
2. 配置规则（白名单/标签/比例）  
3. 发布执行后，仅命中规则流量进入 canary  
4. 观察期结束后：提升为 stable 或执行回滚  

### 6.3 回滚流程

1. 选择某环境当前生效发布单  
2. 回滚到“上一稳定版本”  
3. 写入回滚记录（原因、操作者、前后版本）  
4. 解析服务即时切换为回滚后目标版本  

## 7. 状态机设计

### 7.1 发布单状态（`release_status`）

- `DRAFT`：草稿
- `PENDING_APPROVAL`：待审核
- `APPROVED`：审核通过待执行
- `RELEASING`：执行中
- `SUCCESS`：发布成功
- `FAILED`：发布失败
- `ROLLED_BACK`：已回滚
- `CANCELLED`：取消

### 7.2 转移规则（关键）

- 仅 `APPROVED` 可进入 `RELEASING`
- `RELEASING` 只能流向 `SUCCESS` 或 `FAILED`
- `SUCCESS` 可进入 `ROLLED_BACK`
- 已 `ROLLED_BACK` 的发布单不可再次执行

## 8. 数据模型草案

## 8.1 发布单表 `ai_prompt_releases`

建议字段：

- `id` bigint pk
- `psu_id` bigint
- `environment` varchar(32)（DEV/STAGING/PROD）
- `release_type` varchar(16)（FULL/CANARY）
- `target_composition_id` bigint
- `target_revision_no` int
- `base_revision_no` int（canary 基线版本）
- `status` varchar(32)
- `approval_by` bigint
- `approved_at` datetime
- `executed_by` bigint
- `executed_at` datetime
- `rollback_to_revision_no` int
- `rollback_reason` varchar(500)
- `created_by` bigint
- `updated_by` bigint
- `created_at` datetime
- `updated_at` datetime

关键索引：

- `idx_release_psu_env (psu_id, environment, status)`
- `idx_release_target (target_composition_id, target_revision_no)`

唯一约束建议：

- 同一 `psu_id + environment` 在 `RELEASING` 状态只能有一条

## 8.2 灰度规则表 `ai_prompt_release_rules`

建议字段：

- `id` bigint pk
- `release_id` bigint
- `rule_type` varchar(16)（WHITELIST/TAG/PERCENT）
- `rule_key` varchar(64)（如 tenantId、channel、userId）
- `operator` varchar(16)（EQ/IN/REGEX/RANGE）
- `rule_value` text
- `traffic_percent` int（0-100，PERCENT 生效）
- `priority` int（数值越小优先级越高）
- `enabled` tinyint
- `created_at` datetime
- `updated_at` datetime

关键索引：

- `idx_rules_release (release_id, enabled, priority)`

## 8.3 环境生效指针表 `ai_prompt_live_versions`

建议字段：

- `id` bigint pk
- `psu_id` bigint
- `environment` varchar(32)
- `stable_release_id` bigint
- `stable_revision_no` int
- `canary_release_id` bigint null
- `updated_by` bigint
- `updated_at` datetime

唯一约束：

- `uk_live_psu_env (psu_id, environment)`

说明：

- 解析服务优先看 canary 规则命中，否则返回 stable

## 8.4 回滚记录表 `ai_prompt_rollbacks`

建议字段：

- `id` bigint pk
- `psu_id` bigint
- `environment` varchar(32)
- `from_release_id` bigint
- `from_revision_no` int
- `to_release_id` bigint
- `to_revision_no` int
- `reason` varchar(500)
- `operator_id` bigint
- `created_at` datetime

## 9. API 契约草案

## 9.1 管理侧 API

### 1) 创建发布单

- `POST /api/releases`
- 请求体：

```json
{
  "psuId": 1001,
  "environment": "PROD",
  "releaseType": "CANARY",
  "targetCompositionId": 88,
  "targetRevisionNo": 12,
  "baseRevisionNo": 11
}
```

### 2) 提交发布审核

- `POST /api/releases/{releaseId}/submit`

### 3) 审核发布单

- `POST /api/releases/{releaseId}/approve`
- `POST /api/releases/{releaseId}/reject`

### 4) 执行发布

- `POST /api/releases/{releaseId}/execute`

### 5) 配置灰度规则

- `POST /api/releases/{releaseId}/rules`
- `PUT /api/releases/{releaseId}/rules/{ruleId}`
- `DELETE /api/releases/{releaseId}/rules/{ruleId}`

### 6) 执行回滚

- `POST /api/releases/{releaseId}/rollback`
- 请求体：

```json
{
  "targetRevisionNo": 11,
  "reason": "线上幻觉率上升，紧急回滚"
}
```

### 7) 查询发布详情

- `GET /api/releases/{releaseId}`
- `GET /api/releases?psuId=1001&environment=PROD&page=1&size=20`

## 9.2 对外服务 API（调用方）

### 1) 解析当前可用 Prompt 版本

- `POST /api/prompt-service/resolve`
- 请求体：

```json
{
  "psuId": 1001,
  "environment": "PROD",
  "context": {
    "tenantId": "t-a",
    "channel": "app",
    "userId": "u-100",
    "traceId": "x-123"
  }
}
```

- 响应体：

```json
{
  "psuId": 1001,
  "environment": "PROD",
  "releaseId": 9008,
  "revisionNo": 12,
  "routeType": "CANARY",
  "ruleId": 301,
  "renderConfig": {
    "placeholderStyle": "mustache"
  }
}
```

### 2) 获取指定 revision 的模板详情（内部服务）

- `GET /api/prompt-service/revisions/{compositionId}/{revisionNo}`

## 10. 灰度路由规则（执行顺序）

1. 按优先级读取启用规则  
2. 先匹配 `WHITELIST` / `TAG`，后匹配 `PERCENT`  
3. `PERCENT` 采用稳定哈希（如 `hash(tenantId + userId) % 100`）  
4. 命中任一规则返回 canary，否则返回 stable  

约束：

- 一个请求上下文必须可重复命中同一版本（避免抖动）
- 规则变更需要审计，支持回放

## 11. 一致性与可靠性

- 幂等键：`Idempotency-Key`（创建发布、执行发布、回滚必须支持）
- 乐观锁：发布单执行更新使用版本号或状态条件更新
- 读写一致性：发布执行完成后再刷新 live 指针
- 降级策略：规则服务异常时，回落 stable 版本

## 12. 安全与审计

- 管理侧 API 需角色鉴权（研发/管理员）
- 对外解析 API 使用服务凭证（非用户态 token）
- 所有发布、规则变更、回滚动作写入审计日志

## 13. 验收用例（最小闭环）

### 13.1 发布验收

- 给定审核通过版本，能创建并成功执行 FULL 发布
- 执行后 `live stable revision` 正确更新

### 13.2 灰度验收

- 配置 `PERCENT=10` 后，请求命中比例在统计窗口内接近 10%
- 白名单用户始终命中 canary

### 13.3 回滚验收

- 回滚后解析接口立即返回目标旧版本
- 回滚记录包含前后版本、操作者和原因

### 13.4 异常验收

- 并发执行同一发布单仅允许一次成功
- 非法状态转换返回明确错误码

## 14. 与现有代码的衔接建议

### 14.1 可复用能力

- 复用 `VersionReview` 审核通过结果作为发布输入
- 复用 `PromptCompositionRevision` 作为实际发布内容来源

### 14.2 需要新增能力

- 新增 `ReleaseService`、`ReleaseRuleService`、`PromptResolveService`
- 新增对应 controller/repository/entity/dto
- 在现有“版本审核页”增加“发版中心”入口

## 16. 里程碑完成定义

- 研发可执行灰度并可回滚
- 整个过程可审计、可复盘、可验收
