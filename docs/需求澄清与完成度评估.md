# 需求澄清与完成度评估（V1）

## 1. 文档目标

- 基于当前仓库实现，明确五大模块的需求边界
- 给出“已完成 / 部分完成 / 未完成”清单
- 输出可执行的后续开发规划，减少反复沟通成本

## 2. 范围定义

### 2.1 本次评估模块

1. 提示词模板编辑与调试  
2. 对外提示词服务（查询/版本管理/灰度发布/版本回滚）  
3. JSON Schema 编辑与测试集准备  
4. Prompt 评估（幻觉/上下文连贯性/上下文相关性等）
5. 代码自动生成（Java/Python 调用代码 + Schema 入参组装）

### 2.2 评估口径

- 只认“代码中可执行能力”，不认历史文档描述
- 若后端无强约束、仅前端限制，按“部分完成”处理
- 若功能仅有页面或接口壳但缺主流程，按“未完成”处理

## 3. 模块级需求澄清

## 模块1：提示词模板编辑与调试

### 3.1 目标澄清

- 支持可视化编辑 Prompt 模板
- 支持 Schema 变量选择、插入与校验
- 支持测试集驱动渲染调试与问题定位
- 支持提审后锁定，避免“审核过程内容漂移”

### 3.2 已实现

- 编排草稿保存、校验、渲染、提交审核
- 占位符变量抽取与缺失变量检测
- Schema 字段存在性检查
- 前端编排页支持测试数据集联动预览
- 提交后状态锁定，草稿不可继续修改

### 3.3 部分实现

- 模型真实调试链路不完整：测试运行仍以 Mock 输出为主
- 多角色编辑权限主要依赖页面逻辑，后端鉴权未真正收口

### 3.4 未实现

- 模板历史 diff、回放对比
- 细粒度多人协作（锁、冲突检测、注释讨论）

### 3.5 完成度

- 70%

## 模块2：对外提示词服务（查询/版本管理/灰度/回滚）

### 3.6 目标澄清

- 面向业务系统提供稳定的 Prompt 查询服务接口
- 版本可管理（创建、审核、发布、回滚）
- 支持灰度策略（比例、白名单、标签路由）
- 支持线上快速回滚与可追溯发布记录

### 3.7 已实现

- 版本提交与审核流程（PENDING/APPROVED/REJECTED）
- 编排 revision 快照
- 驳回类型区分（退回业务/退回研发）
- Git hash 登记入口

### 3.8 部分实现

- 版本管理“流程有、发布弱”：仅在审核域内完整，未形成生产生效域
- 代码生成可返回文本，但未形成部署产物与发布绑定

### 3.9 未实现（关键）

- 无“线上生效版本”指针模型（current/live/canary）
- 无灰度发布策略与流量路由器
- 无回滚 API 与回滚审计闭环
- 无对外统一查询协议（按租户/场景/渠道/环境返回版本）

### 3.10 完成度

- 45%

## 模块3：JSON Schema 编辑与测试集准备

### 3.11 目标澄清

- Schema 版本化管理
- 测试集结构化维护
- 使用测试集驱动模板验证和回归

### 3.12 已实现

- Schema 获取、更新、历史版本查询
- 测试数据集 CRUD
- 测试运行主从记录（run + run item）

### 3.13 部分实现

- Schema 语义校验较弱，主要是 JSON 格式可解析校验
- 测试集缺少统一约束（caseId、input、expected 的严格协议）

### 3.14 未实现

- 测试集版本化与冻结机制
- 基线结果管理（golden set）与回归差异报告

### 3.15 完成度

- 75%

## 模块4：Prompt 评估（幻觉/连贯性/相关性）

### 3.16 目标澄清

- 对已发生会话或测试运行结果进行多维评估
- 评估结果可用于版本准入门禁与质量看板

### 3.17 已实现

- 无（仅有测试运行记录，不等于评估）

### 3.18 未实现（关键）

- 评估任务模型与任务执行器
- 评估指标定义与评分协议
- LLM-as-a-judge / 规则评估 / 人工反馈融合
- 评估报告与版本关联、门禁规则

### 3.19 完成度

- 10%

## 模块5：代码自动生成（Java/Python）

### 3.20 目标澄清

- 在提示词通过审核并进入发布流程后，可生成可落地的调用代码
- 至少支持 Java 与 Python 两种语言模板
- 生成结果应包含：接口调用、鉴权头、错误处理、Schema 入参组装与基本校验
- 生成代码应与“发布生效版本”绑定，避免调用方拿到过期模板

### 3.21 已实现

- 后端已有代码生成服务：`CodeGeneratorService`
- 审核通过时可触发代码生成并写入 `ai_prompt_version_reviews.codeContent`
- 已提供查询接口：`GET /api/versions/{psuId}/code`
- 前端已有代码生成页面，可选择 PSU、预览、复制、下载代码文本

### 3.22 部分实现

- 代码生成链路已接入审核流程，但触发点在审核域，未与发布域生效版本强绑定
- 生成逻辑可拼装 Prompt 片段并返回文本，但仍是“示例骨架”而非可直接运行 SDK

### 3.23 未实现（关键）

- 无语言维度参数，未区分 Java/Python 模板与下载产物
- Schema 映射较弱：以 `Object` 和注释占位为主，缺少 DTO/Typed Model 生成
- 无发布维度查询：无法按 `环境 + 生效版本` 获取对应代码
- 无产物治理：无模板版本、生成记录、产物校验与回归测试
- 无最小可运行示例工程（依赖声明、配置示例、调用封装）

### 3.24 完成度

- 40%

## 4. 统一问题清单（一次性梳理同类问题）

> 按“同类型问题一起解决”的原则，以下问题建议成组改造。

### 4.1 版本治理同类问题

- 仅审核域有状态，发布域没有“生效版本”与“回滚状态”
- 无灰度配置，导致上线方式只能全量切换
- 版本号、发布记录、流量策略未统一建模

### 4.2 质量评估同类问题

- 测试运行与质量评估分离不清
- 缺少可解释评分结构，无法给运营/研发可执行反馈
- 无门禁，评估结果无法影响发布决策

### 4.3 安全与治理同类问题

- 认证与鉴权未严格启用，后端接口默认放开
- 角色权限更多在前端体现，缺后端强校验
- 配置与密钥治理需尽快收口（环境隔离、最小权限）

### 4.4 代码生成同类问题

- 生成能力停留在“文本返回”，缺少按语言、按环境、按发布版本的治理模型
- Schema 到入参对象的映射规则未沉淀，调用方仍需手工二次开发
- 生成链路与发布链路分离，存在“生成代码版本与线上生效版本不一致”风险

## 5. 后续开发规划（建议）

## 阶段A：版本治理修正（优先级 P0，2-3 周）

- 新增发布实体：`prompt_release`（环境、版本、状态、生效时间）
- 新增灰度实体：`prompt_release_rule`（规则类型、匹配条件、比例）
- 新增回滚接口：按发布单快速切换到上一稳定版本
- 对外查询接口：`/api/prompt-service/resolve`（入参：场景、租户、标签）

验收标准：

- 可创建发布单并执行灰度
- 可回滚并保留审计日志
- 同一请求上下文可稳定命中同一版本

## 阶段B：评估体系 MVP（优先级 P0，2-4 周）

- 新增评估任务：`evaluation_job`、`evaluation_item`
- 定义评估维度：幻觉、连贯性、相关性、指令遵循
- 输出评分与解释：总分、维度分、证据片段、建议
- 与测试集联动：支持“发布前批量评估”

验收标准：

- 给定测试集可产出完整评估报告
- 报告可回溯到版本与输入样本

## 阶段C：发布门禁与可观测（优先级 P1，2 周）

- 门禁策略：低于阈值禁止发布
- 仪表盘：发布成功率、回滚率、维度评分趋势
- 统一审计：发布/回滚/审核/评估操作可追踪

验收标准：

- 门禁生效且可配置阈值
- 关键指标可视化且可按 PSU 查询

## 阶段D：代码生成落地（优先级 P1，2-3 周）

- 扩展生成接口：支持 `language`（java/python）与 `environment` 入参
- 模板分层：调用封装层、Schema 模型层、错误处理层分离生成
- 与发布域绑定：按生效版本生成并固化产物元数据（PSU、版本号、语言、时间）
- 提供示例工程与回归样例，确保生成代码可直接运行

验收标准：

- 同一 PSU 在 Java/Python 均可生成可运行示例
- 生成代码与发布生效版本一致且可追溯
- 常见 Schema 结构（对象、数组、可选字段）可自动映射

## 6. 建议的需求文档模板（用于下一轮澄清）

为避免再次出现“接口有了但需求未闭环”，后续每个模块建议统一使用以下模板：

1. 业务目标与边界  
2. 用户角色与操作路径  
3. 数据模型（主实体/状态机/约束）  
4. 接口契约（请求、响应、错误码）  
5. 非功能需求（性能、可用性、安全）  
6. 验收标准（正向 + 异常 + 回归）  

## 7. 结论

- 当前项目已具备 Prompt 工程平台雏形，尤其是编排与提审主链路，代码生成入口也已具备。
- 下一阶段必须优先补齐“版本发布域（灰度/回滚）+ 评估域（多维评分）”，并并行推进代码生成落地，否则难以支撑真正生产化。
- 建议先做 P0 的“发布治理 + 评估 MVP”，随后完成 P1 的“代码生成工程化”，再进行 UI/体验扩展，确保核心能力先闭环。
