# 版本号单字段改造方案（V1）

## 1. 结论先行

当前版本号使用 `major/minor/patch` 三字段，确实不适合本项目现阶段业务。  
建议统一改为 **单一版本号属性**：`versionNo`（数据库字段：`version_no`，整型递增）。

## 2. 设计决策

### 2.1 统一规则

- 版本号使用单字段：`versionNo`
- 版本递增策略：每次“可发布语义变更”统一 `+1`
- 不再区分主次修订，不再拼接字符串版本

### 2.2 字段规范

- Java 命名：`versionNo`
- SQL 命名：`version_no`
- API 返回：`versionNo`

## 3. 全量影响面（已盘点）

## 3.1 数据库层

受影响表：

- `ai_prompt_psu`：`major_version/minor_version/patch_version` -> `version_no`
- `ai_prompt_version_reviews`：`major_version/minor_version/patch_version` -> `version_no`
- 唯一约束：`uk_version_reviews_version(psu_id, major, minor, patch)` 需改为 `(psu_id, version_no)`

## 3.2 后端实体/DTO

受影响类：

- `PsuUnit`（3 字段 -> 1 字段）
- `VersionReview`（3 字段 -> 1 字段）
- `PsuResponse`（移除 `majorVersion/minorVersion/patchVersion/fullVersion`，新增 `versionNo`）

## 3.3 后端服务逻辑

受影响类：

- `JsonSchemaService`：原“次版本+1，修订重置”逻辑改为 `versionNo + 1`
- `PromptService`：原“修订+1”逻辑改为 `versionNo + 1`
- `VersionReviewServiceImpl`：提审时从 PSU 拷贝单字段 `versionNo`
- `PsuService`：响应映射从 `fullVersion` 改为 `versionNo`

## 3.4 前端页面

受影响页面：

- `frontend/src/views/developer/PromptManagement.vue`
- `frontend/src/views/developer/Dashboard.vue`
- `frontend/src/views/business/PromptManagement.vue`
- `frontend/src/views/business/Dashboard.vue`

改造方式：

- 所有 `v${majorVersion}.${minorVersion}.${patchVersion}` 改为 `v${versionNo}`

## 3.5 测试

受影响测试：

- `VersionReviewServiceImplTest`
- `PsuServiceTest`

改造方式：

- 断言从三字段改为单字段

## 4. 迁移方案（一次性）

> 建议采用“先加后迁再删”的平滑迁移，避免直接破坏线上数据。

### 4.1 第一步：新增列并回填

在 `ai_prompt_psu` 新增 `version_no`，回填规则：

- `version_no = major_version * 1000000 + minor_version * 1000 + patch_version`

在 `ai_prompt_version_reviews` 新增 `version_no`，按同规则回填。

### 4.2 第二步：应用代码切换

- 应用层全部读写 `version_no`
- 保留旧列只读兼容 1 个迭代周期

### 4.3 第三步：清理旧字段

- 删除旧唯一约束
- 删除 `major/minor/patch` 三列
- 建立新唯一约束 `(psu_id, version_no)`

## 5. 兼容策略

### 5.1 API 兼容（建议 1 个版本窗口）

- 新返回：`versionNo`
- 旧字段可短期保留返回（只读），标注 deprecated
- 下个版本彻底移除旧字段

### 5.2 前端兼容

- 前端统一按 `versionNo` 渲染
- 若后端仍返回旧字段，前端可加兜底：
  `versionNo ?? `${majorVersion}.${minorVersion}.${patchVersion}``

## 6. 验收标准

- 数据库不再存在版本三字段写入
- 后端实体与接口仅使用 `versionNo`
- 前端版本展示全部变更为单字段
- 提审、审核、发布链路版本号可连续递增

## 7. 风险与控制

- 风险：历史数据映射冲突（不同三段映射到同一整数）
- 控制：迁移前先跑冲突检测 SQL；有冲突则人工分配新号

- 风险：前后端同时升级窗口不一致
- 控制：提供 1 个迭代窗口的兼容返回，灰度切换前端

## 8. 推荐执行顺序

1. DB 新增 `version_no` + 回填  
2. 后端实体/服务改造 + 接口兼容  
3. 前端展示与提交链路改造  
4. 联调与回归  
5. 清理旧列与旧逻辑
