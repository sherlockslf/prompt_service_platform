# PSU全生命周期管理平台架构改进建议文档（阶段性更新）

## 1. 概述

本文档基于当前代码基线（Java + Vue前后端项目）进行阶段性分析，识别现存问题并给出可落地改进建议。当前已完成部分基线收敛：PSU/审核单字段版本号、Schema覆盖写、参数集覆盖写、审核预览能力；本文重点关注仍待完善部分。

## 2. 关键问题识别

### 2.0 当前已落地进展（与代码同步）

- PSU与审核主线已统一为单字段版本号（`version_no`）
- JSON Schema已改为覆盖写语义（`psu_id`唯一，`version`为兼容字段）
- 参数集已独立建表并采用覆盖写语义（`ai_prompt_param_sets`）
- 审核链路已支持按参数集进行在线预览
- Redis当前为预留依赖与配置，业务代码尚未启用

### 2.1 安全性问题（高优先级）

#### 2.1.1 密码明文存储（严重安全漏洞）
- **问题描述**：数据库中用户密码以明文形式存储，SecurityConfig使用NoOpPasswordEncoder
- **风险等级**：严重
- **影响范围**：所有用户账户安全
- **根本原因**：缺少基本的安全意识和密码哈希处理

#### 2.1.2 敏感配置硬编码
- **问题描述**：JWT密钥、API密钥等敏感信息直接硬编码在配置文件中
- **风险等级**：高
- **影响范围**：系统整体安全性
- **根本原因**：缺少安全的配置管理机制

#### 2.1.3 认证授权机制薄弱
- **问题描述**：前端仅依赖localStorage存储权限信息，容易被篡改
- **风险等级**：中
- **影响范围**：权限控制有效性
- **根本原因**：前后端权限验证不一致

### 2.2 架构设计问题

#### 2.2.1 分层架构不清晰
- **问题描述**：后端代码结构扁平化，Entity直接暴露给前端，缺少DTO层
- **影响等级**：高
- **影响范围**：代码可维护性和扩展性
- **根本原因**：缺少明确的架构分层规范

#### 2.2.2 权限控制粒度不足
- **问题描述**：仅有角色级别权限，缺少数据级别和操作级别的细粒度控制
- **影响等级**：中
- **影响范围**：业务安全性和灵活性
- **根本原因**：权限模型设计过于简单

#### 2.2.3 数据库设计缺陷
- **问题描述**：完全依赖应用层维护数据一致性，不使用数据库外键约束
- **影响等级**：高
- **影响范围**：数据完整性和一致性
- **根本原因**：对数据库约束机制理解不足

### 2.3 代码质量问题

#### 2.3.1 缺少输入验证和异常处理
- **问题描述**：控制器层缺少参数验证，没有统一的异常处理机制
- **影响等级**：中
- **影响范围**：系统稳定性和用户体验
- **根本原因**：缺少完善的错误处理策略

#### 2.3.2 硬编码和魔法值
- **问题描述**：角色名称、状态值等硬编码在代码中
- **影响等级**：低
- **影响范围**：代码可读性和维护性
- **根本原因**：缺少常量管理和枚举使用

### 2.4 前端架构问题

#### 2.4.1 状态管理不完善
- **问题描述**：Pinia store过于简单，缺少复杂状态管理能力
- **影响等级**：中
- **影响范围**：前端应用复杂度和可维护性
- **根本原因**：前端架构设计不够深入

#### 2.4.2 缺少类型安全
- **问题描述**：使用JavaScript而非TypeScript，缺少类型检查
- **影响等级**：中
- **影响范围**：开发效率和代码质量
- **根本原因**：技术选型考虑不充分

### 2.5 运维和部署问题

#### 2.5.1 配置管理混乱
- **问题描述**：本地配置文件和远程Nacos配置混合使用
- **影响等级**：中
- **影响范围**：部署可靠性和配置一致性
- **根本原因**：缺少统一的配置管理策略

#### 2.5.2 监控和日志不足
- **问题描述**：缺少应用性能监控和集中式日志管理
- **影响等级**：中

## 3. 改进建议

### 3.1 安全性改进（立即执行）

#### 3.1.1 密码安全加固
**目标**：实现安全的密码存储和验证

**具体措施**：
1. 替换NoOpPasswordEncoder为BCryptPasswordEncoder
2. 实现密码迁移脚本，将现有明文密码转换为哈希值
3. 添加密码强度策略和定期更换要求

**代码示例**：
```java
// SecurityConfig.java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // 强度因子12
}

// UserService.java - 密码迁移
public void migratePlainTextPasswords() {
    List<User> users = userRepository.findAll();
    for (User user : users) {
        if (!user.getPassword().startsWith("$2a$")) { // 不是BCrypt格式
            String hashedPassword = passwordEncoder.encode(user.getPassword());
            user.setPassword(hashedPassword);
            userRepository.save(user);
        }
    }
}
```

#### 3.1.2 敏感配置管理
**目标**：安全地管理敏感配置信息

**具体措施**：
1. 移除配置文件中的敏感信息
2. 使用环境变量或Spring Cloud Config Server
3. 集成HashiCorp Vault进行密钥管理

**实施步骤**：
```yaml

  jwt-secret: ${JWT_SECRET:default_secret_for_dev}
  encryption-key: ${ENCRYPTION_KEY:default_key_for_dev}
```

#### 3.1.3 认证授权增强
**目标**：实现健壮的认证授权机制

**具体措施**：
1. 实现刷新令牌机制
2. 添加令牌吊销功能
3. 前端增加服务端权限验证

**架构改进**：
- JWT Token + Refresh Token 双令牌机制
- 可选启用Redis存储活跃会话和吊销列表（当前代码尚未启用Redis）
- 前端每次关键操作都验证服务端权限

### 3.2 架构设计改进

#### 3.2.1 分层架构重构
**目标**：建立清晰的分层架构

**新架构分层**：
```
com.example.psu
├── controller/           # 控制器层（接收请求，返回响应）
│   ├── dto/             # DTO（数据传输对象）
│   └── exception/       # 统一异常处理
├── service/              # 服务层（核心业务逻辑）
│   ├── impl/           # 服务实现
│   └── dto/            # 服务层DTO
├── repository/           # 数据访问层
├── entity/               # 实体类
├── domain/               # 领域模型（DDD）
└── config/               # 配置类
```

**关键改进点**：
1. Controller只处理HTTP相关逻辑
2. Service层包含核心业务逻辑
3. Entity不直接暴露给Controller
4. 使用MapStruct进行对象映射

#### 3.2.2 权限控制精细化
**目标**：实现细粒度的权限控制

**权限模型升级**：
- **RBAC + ABAC混合模型**
- 角色基础权限 + 属性基础权限
- 数据级别权限控制

**实现方案**：

```java
    String value();

// 权限服务
@Service
public class PermissionService {
    public boolean hasPermission(Authentication auth, Long psuId, String permission) {
        // 检查用户是否有权访问指定PSU
        // 结合角色、数据归属、操作类型进行综合判断
    }
}
```

#### 3.2.3 数据库设计优化
**目标**：确保数据一致性和查询性能

**改进措施**：
1. 添加必要的外键约束
2. 优化索引策略，添加复合索引
3. 实现软删除统一处理

### 3.3 代码质量改进

#### 3.3.1 输入验证和异常处理
**目标**：提高系统稳定性和用户体验

**改进措施**：
1. 添加Bean Validation注解
2. 实现统一异常处理
3. 标准化API响应格式

**代码示例**：
```java
// DTO验证
public class PsuCreateRequest {
    @NotBlank(message = "PSU ID不能为空")
    @Pattern(regexp = "^[a-zA-Z0-9_-]+$", message = "PSU ID只能包含字母、数字、下划线和连字符")
    private String psuId;
    
    @NotBlank(message = "名称不能为空")
    @Size(max = 200, message = "名称长度不能超过200个字符")
    private String name;
}

// 统一异常处理
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidationException(MethodArgumentNotValidException ex) {
        List<String> errors = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.toList());
        
        return ResponseEntity.badRequest()
            .body(new ErrorResponse("VALIDATION_ERROR", errors));
    }
}
```

```typescript
  persist: true // 启用持久化
```

## 6. 总结

PSU全生命周期管理平台当前存在多个架构层面的问题，其中安全性问题是最高优先级需要立即解决的。通过本文档提出的改进建议，可以显著提升系统的安全性、可维护性、可扩展性和用户体验。

建议按照实施路线图分阶段推进改进工作，优先解决安全问题，然后逐步完善架构设计和代码质量。在整个改进过程中，要注重测试覆盖和质量保障，确保改进工作不会影响现有业务功能。

通过这些改进，PSU平台将能够更好地支撑业务发展，为用户提供更安全、更稳定、更高效的服务。
